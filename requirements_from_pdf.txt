1. Executive Summary
The proposed application is a secure, simple, dentist-focused patient record management system. It will 
allow approved clinic users to capture patient data from a mobile app and allow clinic 
administrators/doctors to manage users, roles, masters, records, analytics, and compliance controls from a 
web admin panel.
The system should be positioned as a clinic record management and follow-up tool, not as an autonomous 
diagnostic system. Any AI or analytics added in later phases should remain assistive and should not replace 
the clinical judgement of a qualified doctor.
 Mobile app for patient registration, photo capture, X-ray/document upload, prescriptions, reminders, 
and notifications.
 Web admin panel for user approval, role management, medicine master, analytics, audit logs, reports, 
and clinic settings.
 Firebase Authentication, Firestore, Storage, Cloud Messaging, Cloud Functions, and Hosting as the 
proposed backend stack.
 Compliance controls built from day one: consent, privacy policy, role-based access, audit logs, data 
deletion process, Google Play declarations, secure storage, and data minimization.

3. Product Scope
3.1 In Scope - MVP
 Google login and admin approval before access.
 Role-based access control for doctor, admin, receptionist, assistant, viewer, and pending user.
 Patient registration with photo, identity/contact data, medical history, dental details, and consent 
capture.
 Visit history with diagnosis, tooth number, treatment notes, treatment status, and follow-up date.
 X-ray and document upload with document type, visit mapping, and secure storage.
 Prescription creation using medicine master and prescription templates.

 Web admin panel with user approval, role management, patient management, medicine master, 
templates, analytics, audit logs, reports, and clinic settings.

 AI diagnosis, autonomous medical advice, X-ray disease detection, or treatment recommendations 
without doctor approval.
 Payment/billing, insurance claims, WhatsApp messaging, ABHA/ABDM integration, OCR, and multi-clinic 
franchise model.
 Public patient portal unless added after stronger security and legal review.

5. Consent and Patient Data Capture
Consent must be captured before saving medical history, treatment notes, prescriptions, X-rays, documents, 
or patient photos. For minor patients, consent must be captured from a parent or legal guardian.
5.1 Recommended Consent Flow
1. Clinic user opens Add Patient.
2. Patient/guardian reads consent text in English and optional Marathi/Hindi.
3. Patient/guardian ticks an unchecked checkbox confirming that they have read and agreed.
4. Patient/guardian signs using finger/stylus on mobile screen.
5. System stores consent text version, timestamp, captured-by user, patient/guardian details, and signature 
image.
6. System generates an immutable consent record and audit log.
7. Patient record can be saved only after valid consent is captured, except where clinic policy allows 
minimal emergency records.
5.2 Consent Data Fields
Field Purpose
Consent given Confirms acceptance.
Consent type Adult / guardian / digital / written / verbal.
Consent text version Identifies exact text accepted.
Consent text snapshot Stores exact wording shown to patient.
Patient/guardian name and relation Proof of who gave consent.
Timestamp Proof of when consent was given.
Captured by Clinic user who captured consent.
Signature image/PDF Evidence of signed consent.
Withdrawal status Supports future withdrawal/restriction flow.

9. Security and Infosec Requirements
 Authentication: Use Firebase Authentication with Google Sign-In. Do not allow anonymous access to 
patient data.
 Admin approval: Every new user must remain pending until approved and assigned a role.
 Authorization: Use role-based and clinic-based access control in both frontend and backend/security 
rules.
 Least privilege: Each role should get the minimum permissions needed for its job.
 Encryption in transit: All network traffic must use HTTPS/TLS; do not send data over insecure channels.
 Storage security: Patient files must not be publicly accessible. Use private Firebase Storage paths and 
access tokens/rules carefully.
 Audit logs: Track create/update/delete/view/export actions for sensitive records.
 Data minimization: Collect only data needed for treatment, follow-up, and clinic record management.
 Secrets management: No API keys/secrets in public repositories. Use environment configuration and 
restricted Firebase rules.
 Backups: Define backup and restore process for Firestore and Storage.

 Device security: Avoid storing sensitive patient data locally. If offline cache is enabled later, encrypt it.
 Access review: Admin should periodically review users and roles.

 Vulnerability management: Use dependency scanning, static analysis, manual security testing, and

 Who can access: authorized clinic users according to assigned roles.
 Storage and processing: Firebase/Google Cloud or chosen backend providers; describe security

A7. Detailed Compliance, Privacy, and Infosec Functionality
Consent and Notice
 Collect patient consent before storing medical/dental data.
 Provide simple consent language in English and optionally Marathi/Hindi.
 Capture parent/guardian consent for minors.
 Store consent version and exact consent text.
 Allow admin to view consent history.
Access Control
 Admin approval before access.
 Role-based permission enforcement in mobile, web, Firestore, and Storage.
 Clinic-wise data isolation using clinic ID.
 Pending/rejected/suspended users cannot access patient data.
 Super admin control for high-risk actions.
Data Security
 Secure Firebase Authentication.
 Firestore Security Rules for all collections.
 Storage Security Rules for photos, X-rays, prescriptions, and documents.
 No public unrestricted file URLs.
 Encrypt sensitive local cache if offline mode is added.
 Use HTTPS/TLS for all network calls.
Audit and Monitoring
 Log sensitive actions: view, create, edit, delete, upload, download, consent, role change.
 Capture user, timestamp, patient/entity, action, old/new values when possible.
 Protect audit logs from normal edits.
 Review failed access attempts and suspicious activity in future phase.
Google Play Readiness
 Complete Health Apps declaration.
 Complete Data Safety form.
 Publish privacy policy and terms.
 Disclose medical/health data collection and purpose.
 Do not make automated diagnosis claims.
 Provide data deletion/contact process.
Data Lifecycle
 Soft delete/archive sensitive records instead of immediate permanent delete.
 Define retention policy with clinic admin.
 Allow patient correction/deletion request workflow as per clinic policy.
 Allow export of patient file in future phase.
 Backup and recovery plan for clinic data.
